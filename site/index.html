The log-first data layer

Ship features,
not schemas.

You know the drill: design tables, write migrations, build endpoints, add filters, wire up live updates — then the requirements change, and you do it all again. foldbase inverts it. Your app emits immutable events; foldbase folds them into queryable tables, enforces row-level access, and streams changes live. One static binary. Zero per-table code.

The idea in 60 seconds

Store what happened. Derive what is.

A conventional database stores the present — one row per task, overwritten on every change. History is gone the moment you UPDATE. foldbase stores the past: every change is an immutable event appended to a log. The "present" — your tables — is just a fold over that history. And anything derived can be re-derived, which is where all the magic below comes from.

what your app writes — facts

{ "type": "TaskCreated", "payload": { "owner": "alice", "title": "Ship it" } }
{ "type": "TaskMoved",   "payload": { "status": "doing" } }
{ "type": "TaskMoved",   "payload": { "status": "done" } }

No CREATE TABLE

Your tables build themselves

You never write DDL, migrations, or per-table endpoints. You declare your events once — typed, in your own language — and describe how they fold. foldbase infers the columns from the event schema, creates the table, and evolves it when you add fields. That's the whole workflow.

you write this (TypeScript client)

import { FoldBase, defineAggregate, defineProjection } from '@baseworks/foldbase'
import { z } from 'zod'

// 1 · the events — your single source of truth
const Tasks = defineAggregate('task', {
  TaskCreated: z.object({ owner: z.string(), title: z.string() }),
  TaskMoved:   z.object({ status: z.enum(['todo','doing','done']) }),
  TaskDeleted: z.object({}),
})

// 2 · how they fold — columns are INFERRED from field types
const tasks = defineProjection('tasks', Tasks, (on) => ({
  TaskCreated: on.TaskCreated.upsert(e => ({ owner: e.owner, title: e.title, status: 'todo' })),
  TaskMoved:   on.TaskMoved.upsert(e => ({ status: e.status })),
  TaskDeleted: on.TaskDeleted.delete(),
}))

// 3 · register on boot — idempotent, like a migration you never wrote
await fb.putProjection(tasks.def)

foldbase does this — automatically

-- created & evolved for you; never touched by hand
CREATE TABLE read_tasks (
  tenant     TEXT NOT NULL,
  id         TEXT NOT NULL,   -- the stream id (uuidv7)
  owner      TEXT,            -- ← z.string()
  title      TEXT,            -- ← z.string()
  status     TEXT,            -- ← z.enum(…)
  updated_at INTEGER NOT NULL,
  PRIMARY KEY (tenant, id)
);

Rename a field in your event schema and the mapping breaks at compile time — the schema is the one source of truth for validation, wire rules, and columns. Same authoring layer in Python, built on pydantic.

The part a mutable store can't do

New tables are born knowing
everything that ever happened

Six months in, you need a stats dashboard you never planned for. In a state-only database that data simply doesn't exist — you start counting from today. In foldbase you add a projection, replay the log, and the new table appears fully populated from day one:

month 6 — add a projection you never planned

const stats = defineProjection('board_stats', Tasks, (on) => ({
  TaskCreated: on.TaskCreated.inc({ created: 1 }),
}))

await fb.putProjection(stats.def)
// → { ok: true, name: 'board_stats', rebuiltFrom: 8214 }
//                       every event ever, replayed ⤴

and it's already full

The same mechanism makes every view disposable: bad fold rule? Fix it and rebuild. Corrupted table? Delete it and rebuild. The log is the truth; tables are just cache.

One endpoint, every table

Query anything. See only what's yours.

There are no per-table endpoints to build — one generic endpoint queries any registered view, with rich JSON filters. Access is deny-by-default: a row policy must grant it, and tenant isolation is welded into every SQL statement structurally — it can never be forgotten in a policy.

register a row policy once (data, not code)

{ "name": "tasks", "role": "*", "using": "owner = :auth_uid" }

query as alice — sees only alice's rows

const { rows } = await fb.withAuth({ uid: 'alice' })
  .query('tasks', {
    where: { status: { eq: 'doing' } },
    sort:  ['-updated_at'],
    limit: 20,
  })

no identity → no rows. no exceptions.

POST /v1/query/tasks            # without X-Auth-UID
403 { "error": "forbidden" }   # deny-by-default

POST /v1/query/_policies        # internal tables?
404                             # structurally unreachable

"select": ["id; DROP TABLE …"] # hostile input?
400                             # identifiers are whitelisted;
                                # client text never reaches SQL

Even privileged services don't bypass policies — they get an explicit role: "service" policy like everyone else. One rule, zero exceptions.

Live by default

Every append is a broadcast

The log already knows the moment anything changes — so realtime isn't a bolt-on, it's a subscription to the log. Plain Server-Sent Events over HTTP: proxy-friendly, auto-reconnecting, and gap-free — each event carries its position, so a dropped connection resumes exactly where it left off. No message broker to run.

$ curl -sN "$BASE/v1/subscribe?type=task" -H "X-Tenant-ID: demo"

Clients get the same one-liner: fb.subscribe({ type: 'task' }, e => render(e)) in TypeScript (auto-reconnect included), a blocking generator in Python.

Quickstart

The same app, three ways

The full loop — define, emit, query — in raw curl (nothing to install), TypeScript, and Python. Every snippet below is exercised by the project's conformance suite before it lands on this page.

# 0 · run it (dev mode: no auth, in-memory db)
DB_URL=:memory: FOLDBASE_AUTH=none PORT=3001 ./foldbase &
BASE=http://localhost:3001

# 1 · declare a view — data, not code; idempotent like a migration
curl -X PUT $BASE/v1/projections -H "X-Tenant-ID: demo" -d '{
  "name": "tasks",
  "columns": { "owner": "text", "title": "text", "status": "text" },
  "on": {
    "TaskCreated": { "op": "upsert", "set": { "owner": "$.owner", "title": "$.title", "status": "todo" } },
    "TaskMoved":   { "op": "upsert", "set": { "status": "$.status" } },
    "TaskDeleted": { "op": "delete" }
  }
}'
curl -X PUT $BASE/v1/policies -H "X-Tenant-ID: demo" \
  -d '{ "name": "tasks", "role": "*", "using": "owner = :auth_uid" }'

# 2 · emit events (optimistic concurrency: expectedVersion starts at 0)
curl -X POST $BASE/v1/streams/task-1 -H "X-Tenant-ID: demo" -d '{
  "expectedVersion": 0, "streamType": "task",
  "events": [{ "type": "TaskCreated", "streamId": "task-1", "actor": "alice",
               "payload": { "owner": "alice", "title": "Try foldbase" } }]
}'
# → { "events": […], "version": 1, "projected": true }

# 3 · query — the policy scopes rows to the caller
curl -X POST $BASE/v1/query/tasks \
  -H "X-Tenant-ID: demo" -H "X-Auth-UID: alice" \
  -d '{ "where": { "status": { "eq": "todo" } } }'
# → { "rows": [{ "id": "task-1", "owner": "alice", "title": "Try foldbase", "status": "todo" }] }
import { FoldBase, defineAggregate, defineProjection } from '@baseworks/foldbase'
import { z } from 'zod'

// events — typed once, drive everything
const Tasks = defineAggregate('task', {
  TaskCreated: z.object({ owner: z.string(), title: z.string().min(1) }),
  TaskMoved:   z.object({ status: z.enum(['todo', 'doing', 'done']) }),
  TaskDeleted: z.object({}),
})

const tasks = defineProjection('tasks', Tasks, (on) => ({
  TaskCreated: on.TaskCreated.upsert(e => ({ owner: e.owner, title: e.title, status: 'todo' })),
  TaskMoved:   on.TaskMoved.upsert(e => ({ status: e.status })),
  TaskDeleted: on.TaskDeleted.delete(),
}))

const fb = new FoldBase({ baseUrl, token, tenant: 'acme' })
await fb.putProjection(tasks.def)                  // on boot, idempotent
await fb.putPolicy({ name: 'tasks', role: '*', using: 'owner = :auth_uid' })

// emit — payload is validated against the schema before it leaves the process
const write = fb.catalog(Tasks)
const id = Tasks.newId()                           // uuidv7 = the row's PK
await write.emit(id, 0, 'TaskCreated', { owner: 'alice', title: 'Ship it' }, { actor: 'alice' })

// query + live updates
const { rows } = await fb.withAuth({ uid: 'alice' }).query('tasks')
const sub = fb.subscribe({ type: 'task' }, (e) => render(e))
from pydantic import BaseModel
from foldbase import FoldBase, define_aggregate, define_projection

# events — pydantic models are the source of truth
class TaskCreated(BaseModel):
    owner: str
    title: str

class TaskMoved(BaseModel):
    status: str

Tasks = define_aggregate("task", TaskCreated=TaskCreated, TaskMoved=TaskMoved)

# columns inferred from field types; e.owner compiles to "$.owner" —
# and a typo like e.ownerr raises immediately, at author time
tasks = define_projection("tasks", Tasks, lambda on: {
    "TaskCreated": on.TaskCreated.upsert(lambda e: {"owner": e.owner, "title": e.title, "status": "todo"}),
    "TaskMoved":   on.TaskMoved.upsert(lambda e: {"status": e.status}),
})

fb = FoldBase(base_url, token=token, tenant="acme")
fb.put_projection(tasks.definition)
fb.put_policy({"name": "tasks", "role": "*", "using": "owner = :auth_uid"})

write = fb.catalog(Tasks)                      # validated, typed writes
write.emit(Tasks.new_id(), 0, "TaskCreated", {"owner": "alice", "title": "Ship it"}, actor="alice")

rows = fb.with_auth(uid="alice").query("tasks")["rows"]
for event in fb.subscribe(type="task"):       # realtime generator
    handle(event)

Operations

One binary. Your storage. Your box.

foldbase is a single static, CGO-free Go binary (~22 MB) designed to run as a private sibling next to each app — no shared central cluster, no fleet to operate. The blast radius of any credential is one app's own data. Point DB_URL at whichever storage fits; the HTTP surface is byte-identical on all three, proven by the same conformance suite.

Non-negotiables

Three rules the engine never breaks

Everything above is a consequence of three invariants — each one machine-verified by a language-agnostic conformance suite (58 HTTP checks + 13 realtime checks) that gates every change.

01

Log first, always

The append commits before any view updates. A fold failure leaves a view stale — repairable by replay — never a lost or invented fact.

02

Views are disposable

Tables are derived cache. Wipe them, change them, add them years later — the log rebuilds them deterministically, every time.

03

Deny by default, no bypass

No policy → no rows, for anyone. Tenant isolation is AND-ed into every statement structurally; client input can never reach an SQL identifier.

Questions you're probably asking

FAQ

Do I need to know event sourcing?
No. If you can name what happens in your app in past tense — OrderPlaced, TaskMoved, EmailVerified — you already have your events. foldbase handles the rest: storage, ordering, concurrency, folding, querying. The theory is optional; the benefits aren't.
Do I need Kafka, or any message broker?
No. foldbase runs one instance per app, which means one process and one writer — the realtime fan-out is in-process, and ordering comes from the log itself. There is nothing else to deploy, configure, or babysit.
What happens when my schema changes?
Add fields to your event schema — new columns appear additively. Need a different shape entirely? Register a new projection and rebuild: it's computed from the full history, so it launches complete, not empty. You never write a migration and never lose data, because the source of truth (the log) never changed shape in the first place.
How do I fix wrong data if events are immutable?
The same way accounting has for centuries: a correcting entry. Emit a new event (TaskRetitled, OrderAmended) and the fold takes care of the current state. The mistake stays in history — which is exactly what you want for auditability.
Won't the log grow forever?
It grows append-only, which is the cheapest, most compressible growth pattern a database can have — no page splits, no rewrite churn. Events are small JSON facts; SQLite and Postgres both handle many millions comfortably, and reads never scan the log (they hit the folded tables). Since each app has its own instance, one app's history never weighs on another's.
Can browsers talk to foldbase directly?
By design, no — foldbase is an internal sibling. Your app verifies its users and forwards their identity (X-Auth-UID); row policies do the scoping. This keeps the trust model simple and the credentials server-side. Direct user-token query support exists (user-jwt mode) for when you want it.
What exactly is it built on?
A single Go binary — no ORM, no framework, raw parameterized SQL behind a strict identifier whitelist. Storage is embedded SQLite, networked libsql/Turso, or PostgreSQL. Behavior is pinned by an OpenAPI contract and a conformance suite that runs against real servers over HTTP — the docs on this page are backed by those tests. TypeScript and Python clients ship today; the API is plain HTTP, so any language works.
