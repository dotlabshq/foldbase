eventstore-service — the log, its views, and the way to query them

An append-only event log over HTTP, plus its materialized read models and one generic query surface. The universal data-access layer for every project — each app runs its own private instance.

The pipeline

Log first, always

Events are the truth. Read models are derived, disposable views folded from the log by declarative rules — never an input to event production.

01 · write
append(event)events table — append-only, optimistic concurrency, global order
──▶
02 · fold
apply rulesdeclarative _projections — upsert / inc / delete, after commit
──▶
03 · materialize
read_<name>derived tables, one row per stream, disposable
──▶
04 · read
POST /v1/query/:namegeneric, policy-gated, tenant-scoped
◆ the iron rule The fold runs after the append commits. A fold failure can only leave a projection stale (the response carries projected:false, repaired by POST /admin/rebuild) — it can never invent or lose a fact.

Table taxonomy

Three kinds of table, three lifecycles

The naming is load-bearing. The query engine resolves names only through the registry, so _-prefixed physical tables are structurally unreachable from client input.

events
The log — the single source of truth. Immutable facts, corrected only by appending new events.
append-only · sacred
_projections · _policies
Definitions as data, pushed by the owning app on its boot like migrations. Never queryable.
control-plane managed
read_<name>
Derived read models. Wiped and refolded from the log on demand — the log is authoritative.
disposable · rebuild anytime

Trust boundary

Auth is declared, and fails closed

The mode is set explicitly via EVENTSTORE_AUTH — a secured mode with no secret, or production with no mode, refuses to boot. No silent fallback to an open service.

Row access is deny-by-default with no bypass for anyone — a service reading without an end-user identity registers an explicit role:"service" policy. Tenant isolation is structural: tenant = ? is AND-ed into every query and every fold, never dependent on policy text.

Wiring an app

Definitions on boot, then append & query

The owning app holds the service token and forwards the already-verified end-user identity. Same client shape in every language.

TypeScript · @baseworks/eventstore-client
// on boot — idempotent, like migrations (control plane)
await es.putProjection({
  name: 'tasks', columns: { owner: 'text', status: 'text' },
  on: { TaskCreated: { op: 'upsert', set: { owner: '$.owner', status: 'todo' } } },
})
await es.putPolicy({ name: 'tasks', role: '*', using: 'owner = :auth_uid' })

// write — optimistic concurrency; 409 ⇒ re-read & retry
await es.append('t1', 0, [{ type: 'TaskCreated', streamId: 't1',
  actor: 'alice', payload: { owner: 'alice' } }])

// read — forward the end-user; the owner policy scopes the rows
const { rows } = await es.withAuth({ uid: 'alice' })
  .query('tasks', { where: { status: { eq: 'todo' } }, sort: ['created_at'] })

The Python client is the same surface, dependency-free (stdlib only): es.put_projection(...), es.append(...), es.with_auth(uid=...).query(...).

Contract-first · polyglot

One contract, many implementations

Behavior is defined by openapi.yaml and locked by a language-agnostic conformance suite that boots any server binary and asserts the contract over HTTP. Every implementation must green the same 49 checks — that is what makes the Go rewrite safe and every client faithful.

Roadmap

Toward Supabase parity

  • Client schema catalog — typed event schemas drive read-model columns & typed emit (designed, pending)
  • UUIDv7 event ids — time-ordered inserts on the id index (proposed)
  • Realtime — SSE / webhook off the append hook; the headline feature
  • Batch & atomic rebuild — streamed replay, shadow-table swap for scale
  • PostgreSQL dialect — behind database/sql, invisible to every client
  • EdDSA + kid rotation — asymmetric verification, no shared secret to distribute
